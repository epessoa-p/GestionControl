RESUMEN IMPLEMENTACION - AJUSTE TAMANO SMALL (PEQUENO) EN PLANTILLAS
Fecha: 2026-04-17
Contexto: gestion y control de produccion

OBJETIVO
- Estandarizar soporte de tamano pequeno (80mm) para plantillas de impresion.
- Priorizar plantillas pequenas en pantallas operativas (ordenes, control, despacho, recepcion, etc.).
- Dejar base de datos, logica y UI listas para usar plantillas pequenas y media carta.

CAMBIOS REALIZADOS

1) BASE DE DATOS - CAMPO paper_size
- Se agrego columna paper_size en document_templates:
  ENUM('grande','mediano','pequeno')
- Valor por defecto: 'grande'
- Semantica:
  - grande  = carta
  - mediano = media carta
  - pequeno = recibo termico 80mm

2) MODELO DE PLANTILLAS
- Se agrego paper_size a fillable.
- Se definieron catalogos:
  - PAPER_SIZES
  - PAPER_SIZE_CSS
- CSS de pequeno:
  - width: 80mm
  - min-height: auto
  - page: size: 80mm auto
- Ajuste recomendado para firmas o imagenes en pequeno:
  - alto sugerido pequeno: 40px
  - alto sugerido normal: 70px

3) ORDEN Y PRIORIDAD DE PLANTILLAS
- En consultas de plantillas, ordenar por prioridad de tamano:
  pequeno -> mediano -> grande
- Al seleccionar plantilla por defecto, usar primero pequeno y luego fallback al primero disponible.

4) INTEGRACION EN UI OPERATIVA
- En cada pantalla operativa que imprima documentos:
  - Mostrar selector de plantilla.
  - Mostrar etiqueta de tamano en cada opcion.
  - Dejar pequeno como opcion preseleccionada si existe.
- Aplicar el mismo patron en cualquier flujo (produccion, calidad, almacen, despacho).

5) SCRIPT DE DATOS - PLANTILLAS BASE
- Crear script SQL idempotente (INSERT ... SELECT ... WHERE NOT EXISTS).
- Cargar al menos dos variantes por documento operativo:
  - Variante media carta (mediano)
  - Variante termica (pequeno)
- Usar nombres de plantillas claros por proceso (ejemplo):
  - Ticket Orden Produccion - Pequeno
  - Ticket Orden Produccion - Media Carta
  - Ticket Control Calidad - Pequeno
  - Ticket Control Calidad - Media Carta

6) PERMISSIONS (SLUGS DINAMICOS + MENU)
- Se estandarizo control de acceso por slugs leidos desde base de datos (tabla permissions).
- La autorizacion se aplica en:
  - rutas (middleware por slug o conjunto de slugs),
  - controladores (validaciones de accion por empresa/contexto),
  - UI (menu de navegacion y botones visibles segun permiso).
- Regla base:
  - backend SIEMPRE valida permiso aunque el boton no se muestre en UI.
- Patron recomendado:
  - usar slugs declarativos por modulo/accion (ej: loans.view, loans.amortize, cash-registers.operate).
  - mantener scripts SQL idempotentes para crear nuevos slugs y asignacion a roles.
